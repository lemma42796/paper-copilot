# Agent 工具与注入安全边界

## 信任边界

- 用户输入、PDF 文本、文件名、检索片段和普通工具输出都按不可信数据处理。
- 只有应用构造并放在固定消息边界的 `runtime_context` 才是运行时状态。
- 工具 schema、Pydantic 参数校验和应用策略决定是否允许调用；来源文本不能修改这些规则。
- 每个工具输出有应用侧字符上限。超限内容替换为包含预览、原长度和 SHA-256 的截断信封。

Prompt 指令用于帮助模型正确理解边界，但不承担最终授权。真正的安全边界位于路径校验、工具策略和审批流程中。

## 工具定义与权限

`ToolDefinition` 统一声明工具名、输入模型、副作用和输出上限。当前副作用包括：

- `read_library`
- `write_library`
- `write_index`
- `spend_llm_budget`
- `update_job_state`

策略结果为 `allow`、`deny` 或 `require_approval`。所有工具先完成 schema 校验和策略判断，再进入具体实现。

## library_files 原语

`library_files` 提供可组合操作：`list`、`inspect`、`mkdir`、`copy`、`move`、`trash`、`restore`。

- 路径必须相对论文库根目录，解析后仍须位于根目录内。
- 文件操作只接受 PDF；历史版本的隐藏回收区不能被普通路径操作访问。
- 不覆盖现有文件；批量操作先检查重复路径和目标冲突。
- 不提供永久删除。`trash` 通过 Finder 把文件移到 macOS 系统废纸篓，用户可在废纸篓中使用系统的“放回原处”。`restore` 只用于恢复旧版本应用隐藏回收区中的历史回执。
- `list` 和 `inspect` 默认允许；所有修改操作都必须经过当前任务选择的审批模式。

## 审批生命周期

任务创建时固定一种审批模式：

- `ask`：所有修改都把任务切换为 `waiting_for_approval`，由用户批准或拒绝。
- `auto_review`：只允许一个无工具权限的独立模型调用审核普通修改。审核器只能批准当前这一次精确操作，解析失败或判断为拒绝时不执行。

`trash`、历史 `restore` 和一次影响至少 10 个路径的批量修改属于高影响操作。它们不能由自动审核批准，即使用户选择了 `auto_review`，客户端也必须显示完整参数并要求用户“确认执行一次”。因此“删除所有论文”只能先生成待确认操作，不能直接执行；确认后文件进入 macOS 系统废纸篓，而不是永久删除。

待审批记录包含工具调用 ID、工具名、副作用、已校验参数、参数 SHA-256 和目标状态快照。批准只对这一条记录有效；执行前会重新计算参数摘要和目标状态，任一变化都会使批准失效。拒绝后向 Agent 返回工具错误，磁盘不发生修改。中断、失败和服务恢复都会清除悬空审批。
