# Paper Copilot 面试亮点：论文研究 Agent 工具设计

> 记录日期：2026-07-21
> 当前状态：核心工具接口、确定性校验和单元测试已实现；真实 LLM 的工具选择正确率、任务完成率、token、成本和延迟尚未量化。

## 一句话定位

围绕研究生和教师的真实论文研究流程，设计并实现一组面向用户意图的 Agent 工具：统一论文定位，区分论文发现、全文读取、单篇问答、多篇比较和关系扩展，并用强类型 Schema、确定性消歧、证据引用和论文预算约束模型行为。

推荐名称：**论文研究 Agent 工具设计**。

## 问题与约束

用户并不会用“向量召回”“chunk 数量”或“知识图谱遍历”描述需求，而会直接提出：

- “帮我讲一下这篇论文”，输入可能是标题、本地 PDF 路径或已有论文 ID；
- “论文为什么这样设计？消融实验能证明什么？”；
- “哪些论文用了同一个 baseline 或数据集？”；
- “比较这些论文的方法、创新点、实验和局限”；
- “这篇论文基于哪些工作，又被哪些工作比较过？”

原有工具更接近内部实现步骤，例如分别列出已索引论文、本地 PDF 和检索结果，并要求模型先检查论文再决定下一步。这样会把底层存储状态和 RAG 参数暴露给模型，增加选错工具、重复调用和参数无效的机会。

设计同时受以下约束：

- 本地 PDF、结构化论文信息和索引状态可能不一致；
- 标题可能只匹配到一部分，也可能对应多篇论文，不能静默猜测；
- 单篇精确问答需要原文证据，结构化筛选不应为了“统一”而强制走 RAG；
- 比较多篇论文前必须确认所有目标，不能边解析边消耗论文预算；
- 工具输出必须支持引用、缺口说明和后续追问，而不只是返回一段自然语言；
- 不增加新的依赖和 LLM 调用，复用已有 Hybrid RAG、结构化论文库和关系数据。

## 关键设计决策

### 1. 工具按用户意图命名

核心工具收敛为五类能力：

| 工具 | 用户意图 | 主要输出 |
|---|---|---|
| `search_papers` | 找论文、按数据集或 baseline 筛选 | 候选论文、匹配证据、索引缺口 |
| `read_paper` | 首次读取或导入一篇论文 | 结构化摘要、可继续引用的证据 |
| `query_paper` | 针对一篇论文追问 | 限定该论文的原文证据与引用 |
| `compare_papers` | 比较 2–5 篇论文 | 指定维度的并列信息、共同项和差异 |
| `find_related_papers` | 沿论文关系扩展阅读 | 指定方向和关系类型的候选论文 |

`list_papers`、`list_pdfs`、`search_library` 和 `inspect_paper` 不再作为模型可见工具。它们表达的是内部操作，不是用户最终想完成的任务。

Composer 的三个专用工具暂时保留，但不继续扩展；该能力后续可能改为 ReAct、Plan-and-Execution 或多 Agent 架构，因此本次没有把核心研究工具绑定到 Composer 流程。

### 2. 用统一 Paper Locator 消除参数漂移

所有需要指定论文的工具都接收相同的 `PaperLocator`，且只允许以下三种定位方式之一：

```json
{"paper_id": "paper-123"}
{"title": "Attention Is All You Need"}
{"pdf_path": "/path/to/paper.pdf"}
```

Pydantic 生成的 JSON Schema 使用互斥 union，并禁止额外字段。这样模型不需要记住每个工具分别使用 `id`、`paper_title` 还是 `path`，参数错误能在工具边界被拒绝。

标题解析采用确定性顺序：标准化后的完整匹配优先，其次才是包含匹配；若存在多个候选，返回候选列表并要求消歧，绝不自动选择“最像”的一篇。

### 3. RAG 问答与结构化查询分开

`query_paper` 在需要基于原文回答时使用已有 Hybrid RAG，并把检索范围强制限定到目标论文。模型只需要描述问题和证据条数，不接触 `k`、候选池大小、chunk 数量等实现参数。`search_papers` 也可以用混合检索发现候选，但不会把候选片段直接当作针对某篇论文的完整答案。

以下任务不强制走 RAG：

- 按年份、会议、方法、数据集或 baseline 筛选论文；
- 展示已经落盘的结构化摘要；
- 比较多篇论文已有的结构化字段；
- 遍历已经建立的论文关系。

当论文没有索引或 embedding 不可用时，工具明确返回 `structured_only`、`needs_read` 或其他缺口状态，而不是用结构化摘要伪装成原文检索结果。

### 4. 搜索接口隐藏底层存储差异

`search_papers` 用 `scope = indexed | local | all` 表达用户希望搜索的范围，并支持年份、会议、方法、数据集和 baseline 等嵌套过滤条件。

内部根据任务选择路径：有查询且索引可用时复用向量检索、BM25 和 RRF；embedding 不可用时降级为标题与结构化字段匹配；未索引的本地 PDF 只能按文件名发现，并在结果中标明能力缺口。

### 5. 多篇比较先解析、后占预算

`compare_papers` 支持 2–5 篇论文，并允许只比较 `contributions`、`methods`、`experiments`、`limitations` 中用户关心的维度。

工具先解析全部 locator，再统一检查歧义、缺失、未读取和重复项。只有所有论文都可以确定时才占用论文分析预算，避免一个失败的多论文请求留下部分状态。成功后基于已落盘的结构化信息构造共同方法、数据集、baseline 以及逐篇差异；两篇论文时额外返回有上限的对齐结果。

### 6. 关系发现与论文分析预算分离

`find_related_papers` 支持 incoming、outgoing 和 both 三种方向，以及 `builds_on`、`compares_against`、`applies_in_different_domain`、`shares_method`、`contrasts_with` 五类关系。

发现候选论文本身不计入“已分析论文”预算；只有后续真正读取、提问或比较时才消耗预算。这使广度探索和深度分析具有不同的成本语义。

## 实现范围

- `agents.paper_copilot`：核心工具 Schema、统一论文定位、确定性消歧、工具调度、搜索/RAG 降级状态和预算控制；
- `knowledge.compare`：多论文、按维度生成比较 payload，并限制每个维度的输出规模；
- `agents.composer_plan`：让现有 Composer 流程改用读取/提问语义，不再依赖已移除的检查工具；
- `tests.agents.test_paper_copilot`：覆盖 Schema、标题歧义、搜索范围、单论文检索、多篇比较、关系过滤和预算行为；
- `ARCHITECTURE.md`：同步核心工具链描述。

本次没有新增依赖，也没有增加 LLM 调用点。`query_paper` 复用现有检索链路，`compare_papers` 和 `find_related_papers` 基于已有结构化数据执行。

## 主要取舍

- 没有设计一个万能 `research` 工具。过大的工具会把发现、读取、检索和比较混在同一参数空间中，Schema 很难清晰表达失败状态。
- 没有让所有问题都走 RAG。结构化筛选和关系遍历使用确定性数据更便宜，也更容易解释。
- 没有为模糊标题自动打分并选择第一名。多一次用户消歧优于分析错论文。
- 没有把 Composer 一并重新设计。Composer 的未来架构尚未确定，当前只保证它能够使用新的核心工具语义。
- 没有新增一个 LLM 负责路由。唯一 Agent 继续根据工具 Schema 选择能力，代码只执行必须成立的校验。

## STAR 面试讲述

### Situation

论文 Agent 的工具直接暴露了“列索引”“列 PDF”“检索库”“检查论文”等内部步骤。用户意图需要跨多个工具映射，模型还要理解索引状态和 RAG 参数，容易产生重复调用、错误参数或分析错论文。

### Task

在保持单 Agent tool loop、不增加依赖和 LLM 调用的前提下，设计一组能覆盖论文发现、精读问答、多篇比较和关系探索的清晰工具，并保证歧义、证据缺口和预算约束可由代码验证。

### Action

1. 从研究生和教师的典型提问反推五类用户意图；
2. 为所有工具统一 `PaperLocator`，用互斥 JSON Schema 限制输入；
3. 把标题匹配、歧义返回和论文预算放入确定性代码；
4. 只在原文问答中启用限定单篇论文的 Hybrid RAG；
5. 将结构化筛选、多篇比较和关系发现保持为确定性路径；
6. 增加 Schema、状态机、检索范围、比较维度和预算行为的回归测试。

### Result

核心工具面收敛为五类用户语义能力，支持 ID、标题和本地路径统一定位；模糊标题不再被静默选择，多论文请求不会在解析失败时部分消耗预算。全套 246 项单元测试通过，且未新增依赖或 LLM 调用。真实 LLM 的工具轨迹和成本收益仍待多轮评测。

## 可能的面试追问

### 为什么工具越少不一定越好？

目标不是最少数量，而是每个工具对应稳定、可区分的用户意图。若把搜索、阅读、问答和比较合成一个工具，输入 Schema 会出现大量条件字段，失败状态也难以解释；拆得过细又会迫使模型规划内部步骤。五个核心工具是在意图完整性和决策复杂度之间的折中。

### 为什么统一 Paper Locator 很重要？

用户输入论文的方式天然不同，但这不应该让每个工具形成一套参数协议。统一 locator 能降低模型记忆负担，并让消歧、路径校验和未读取状态复用同一套确定性逻辑。

### 什么情况下使用 RAG？

当答案必须回到原文定位证据时使用，例如方法细节、公式解释、消融结果和局限性。若答案来自已经结构化的年份、数据集、baseline、方法标签或论文关系，则直接查询结构化数据，避免额外召回噪声。

### 为什么 `query_paper` 还允许模型指定 `evidence_limit`？

它表达的是用户答案需要多少条可展示证据，而不是底层召回算法参数。候选池和 chunk 检索规模由系统固定，避免模型参与检索调参。

### 为什么不让模型直接处理标题歧义？

语言模型给出的“最可能”不是身份确认。论文分析成本较高，且错误论文会污染后续比较和引用，因此完整匹配、包含匹配与候选返回由代码执行；无法唯一确定时必须让用户消歧。

### 为什么比较工具不再调用一次 LLM？

已有论文读取流程已经生成结构化贡献、方法、实验和局限。比较工具先做字段对齐和共同项计算即可给主 Agent 足够材料，最终叙述仍由主 Agent 完成；额外 LLM 调用会增加成本，也可能引入二次改写偏差。

### 如何证明这套工具设计真的更好？

单元测试只能证明契约和确定性约束正确，不能证明模型更会选工具。需要固定任务集，比较设计前后的首次选择正确率、无效参数率、重复调用、完成轮数、证据覆盖率、成本和延迟，并对随机模型输出做多轮趋势观察。

## 尚未完成与不可宣称内容

- 尚未建立工具设计前后的真实 LLM 对照基线；
- 尚未统计研究生、教师等真实用户的任务完成率；
- 尚未验证工具描述在不同模型上的迁移效果；
- 尚未量化 token、成本、延迟或 Prompt cache 的变化；
- Composer 的 ReAct、Plan-and-Execution 或多 Agent 方案尚未决定，也未实现。

面试时可以明确宣称“工具体系、Schema、确定性约束和测试已经实现”，不能把待测指标表述为已经取得的效果。
