# Paper Copilot 面试亮点：长任务 Agent 上下文压缩

> 记录日期：2026-07-21  
> 当前状态：核心机制、自动接入、专项测试和一次真实模型受控评测已完成；真实多轮
> Composer 的压缩前后 A/B 质量评测因时间限制未执行。面试中不得宣称最终报告质量
> “无下降”。

## 一句话定位

为长任务论文 Agent 实现 token 驱动的上下文压缩机制：在保留原始请求、权威运行
状态、完整工具轮次和证据标识的前提下，用结构化 LLM 摘要压缩旧历史，并通过
确定性校验、重试、紧急门槛和 append-only 审计降低摘要失真风险。

## 为什么这不是普通的“聊天记录总结”

普通聊天摘要只要语义大致连贯即可；Agent 历史还包含会影响后续执行的状态：

- 用户原始目标和不可变约束；
- 已选择的 baseline 和 accepted modules；
- 已拒绝的候选及失败原因；
- 论文 ID、evidence ref 等必须精确保留的标识；
- tool-use 与 tool-result 的协议配对；
- 当前论文预算、成本预算和 Composer 流程状态；
- 尚未解决的问题和下一步动作。

摘要即使读起来合理，只要丢失一个 rejected decision，Agent 就可能重新采用已经否决
的方案；只要截断一半工具轮次，下一次模型请求就可能成为非法协议。因此问题的核心
不是“压得更短”，而是“压缩后还能安全地继续执行”。

## 目标与约束

项目采用以下上下文策略：

| 参数 | 数值 | 作用 |
|---|---:|---|
| 模型标称窗口 | 1,000K tokens | 模型能力上限，不作为日常工作预算 |
| 工作窗口 | 256K tokens | 项目主动控制的输入上限 |
| 自动压缩触发点 | 200K tokens | 为生成、工具结果增长和估算误差预留空间 |
| 压缩后目标 | 不超过 80K tokens | 重新获得足够的后续工作空间 |
| 近期原文预算 | 40K tokens | 保留最近的完整推理和工具交互 |
| 紧急门槛 | 240K tokens | 无可靠压缩结果时禁止继续普通调用 |
| 摘要最大输出 | 8K tokens | 限制压缩调用的成本和摘要膨胀 |

这些数字不是模型物理极限。即使模型支持 1M，上下文越长仍会带来更高成本、延迟、
注意力稀释和后续增长空间不足。项目选择在 200K 主动治理，而不是等到 API 报错。

## 核心设计

### 1. 阈值前发送原始历史

未达到 200K 时不做摘要，模型直接看到原始消息。这样短任务没有额外 LLM 调用，
也不会过早损失细节。

主 loop 记录上一轮 API 返回的真实输入 token 高水位，再结合此后新增历史的本地估算
计算下一轮输入规模。相比只按消息条数触发，这能适应长度差异很大的论文工具结果。

### 2. 只在完整工具轮次边界切分

压缩器把以下结构视为不可拆分的 round：

```text
assistant: tool_use(id=X)
user:      tool_result(tool_use_id=X)
```

近期历史只能按完整 round 保留。输入中如果已经存在未配对、重复或错配的工具调用，
压缩器直接拒绝处理，不用摘要掩盖协议损坏。

### 3. 分离“必须原样存在”和“允许语义压缩”的信息

压缩后的模型输入由四部分构成：

```text
原始用户请求
+ 旧历史的结构化 CompactionSummary
+ 最新权威 Runtime Context
+ 近期完整工具轮次原文
```

其中：

- 原始请求重新注入，防止任务目标被多级摘要漂移；
- Runtime Context 由应用代码重新生成，是预算与 Composer 状态的权威来源；
- 最近 40K 保持原文，让模型继续处理当前局部任务；
- 只有更旧的历史交给 LLM 结构化压缩。

旧的 Runtime Context 不进入保留区，避免过期预算或流程状态与最新快照冲突。

### 4. 使用结构化摘要，而不是自由文本

压缩模型必须通过 forced tool call 返回 `CompactionSummary`：

- `original_goal`
- `active_constraints`
- `decisions`
- `completed_work`
- `evidence_and_identifiers`
- `failed_attempts`
- `open_questions`
- `next_actions`
- `superseded_information`

结构化字段让遗漏更容易检测，也明确要求保留失败尝试和被取代的信息，避免摘要只记录
“成功故事”。

### 5. LLM 负责语义压缩，代码负责安全不变量

项目没有假设 LLM 摘要天然可靠。每次摘要后执行确定性校验：

- 运行状态收集出的必需论文 ID、决策标识和 evidence refs 必须仍然出现；
- 摘要不得凭空生成旧历史中不存在的 evidence ref；
- 压缩后的估算输入不得超过 80K；
- 工具调用轮次必须继续完整配对。

校验失败时，把具体错误作为可信 retry 反馈，最多重试一次。两次仍失败则抛出错误，
不带着不可信摘要继续执行。

### 6. 重复压缩采用增量摘要

会话再次达到阈值时，上一版结构化摘要与新增旧历史一起送入压缩模型。这样无需恢复已经
被压缩过的原文，也避免每次从头总结全部会话。

摘要带 `version`，为将来的 Schema 演进和历史兼容留出空间。

### 7. 原始 Session 永不删除

压缩只改变“下一次发给模型的工作上下文”，不删除 `session.jsonl` 中的原始消息。
每次压缩追加一条 `compaction` 记录，包括：

- 压缩前后估算 token；
- 来源消息数和近期保留消息数；
- 近期原文 token；
- 摘要输出 token；
- 模型与摘要版本；
- 完整结构化摘要。

因此线上工作上下文可以变短，但问题复盘、成本核算和审计仍能访问完整原始轨迹。

## 安全执行流程

```text
估算下一轮 input tokens
  ├─ < 200K：发送原始历史
  └─ ≥ 200K：检查工具轮次完整性
                ↓
             保留近期完整 rounds
                ↓
             LLM 生成结构化摘要
                ↓
             确定性检查标识与证据
                ├─ 失败：带错误重试一次
                └─ 成功：重建最新 Runtime Context
                              ↓
                         检查压缩后 ≤ 80K
                              ↓
                         追加 compaction JSONL
                              ↓
                         继续 Agent loop

任何路径接近 240K 且没有可靠压缩结果：停止普通模型调用
```

## 实现范围

- `agents/llm_client.py`：集中定义窗口、触发点、目标、近期预算和紧急门槛；
- `schemas/compaction.py`：结构化 `CompactionSummary`；
- `agents/context_compaction.py`：轮次切分、摘要调用、确定性校验、重试和重建历史；
- `agents/loop.py`：真实 token 高水位、下一轮估算、自动触发和紧急门槛；
- `agents/paper_copilot.py`：接入 Composer 状态、必需标识和增量摘要；
- `agents/tool_validation.py`：为 forced tool call 增加业务校验与逐次费用记录；
- `session/types.py`、`session/store.py`：append-only compaction 记录；
- `scripts/eval_context_compaction.py`：真实 qwen3.6-flash 受控长上下文评测；
- `tests/agents/test_context_compaction*.py`：压缩核心和 loop 边界测试。

没有引入新依赖，也没有删除或改写原始会话数据。

### 真实模型受控评测

使用 qwen3.6-flash 对一段确定性构造的 Composer 风格长历史执行真实压缩：

| 指标 | 结果 |
|---|---:|
| 压缩前估算输入 | 223,704 tokens |
| 压缩后估算输入 | 22,859 tokens |
| 输入缩减 | 约 89.8% |
| 摘要输出 | 872 tokens |
| 来源消息 | 26 |
| 近期保留消息 | 2 |
| 单次压缩成本 | ¥0.24626 |

受控断言确认摘要保留了：

- 原始研究目标；
- `max_papers=5` 和 `budget_cny=2.0`；
- baseline 与三个 module ID；
- 四个 evidence refs；
- 已拒绝的 reranker 决策及原因；
- 未解决的兼容性问题；
- 下一步生成研究方案的动作。

准确表述应是：**受控样本中的必需锚点零遗漏**，不能扩大为“所有上下文细节零丢失”。

## 未完成的验证

没有执行真实多轮 Composer 的压缩前后 A/B 测试。原因是自然会话需要多轮交互才可能
跨过 200K，并且需要多次运行才能把模型随机噪声与压缩影响区分开，当前时间成本不合适。

因此尚未证明：

- 压缩前后最终 Composer 报告质量无下降；
- baseline 和 module 选择趋势完全一致；
- 真实长任务中的平均压缩频率；
- 多次增量压缩后的长期语义漂移上限；
- 端到端成本和延迟收益。

这是有意识地缩小验证范围，不应用受控摘要测试代替端到端质量结论。

## 主要取舍

### 为什么不把全部 1M 窗口用满？

模型能接收不代表系统应该持续发送。长输入会增加成本与延迟，也给后续工具结果和输出
留下更少空间。256K 是项目工作预算，1M 是最终物理边界。

### 为什么不用滑动窗口直接丢弃旧消息？

旧历史包含决策、失败路径和证据来源。直接丢弃虽然便宜，但 Agent 可能重复失败尝试，
或无法解释当前状态是如何形成的。

### 为什么不用纯代码摘要？

论文研究过程包含自然语言理由、兼容性判断和开放问题，固定规则难以完整抽取。LLM 适合
做语义压缩，但精确标识和协议不变量仍交给代码验证。

### 为什么保留最近 40K 原文？

Agent 当前正在处理的局部任务对细节最敏感。保留近期原文可以减少摘要立刻介入当前推理
链路，同时让旧历史承担主要压缩比例。

### 为什么失败信息必须单独成字段？

摘要模型容易偏向最终结论。把失败尝试作为一等字段，可以降低 Agent 忘记已验证失败路径、
重复调用和重复花费的风险。

### 为什么压缩失败要停，而不是降级为截断？

静默截断可能让系统继续产生看似正常但状态错误的结果。显式失败更容易被用户发现、重试
和审计，符合“raise early, catch late”的工程原则。

## STAR 面试讲述

### Situation

论文研究 Agent 会连续检索、阅读、比较和组合多篇论文。原始工具结果不断累积，即使模型
支持 1M 上下文，也会产生费用、延迟和状态注意力稀释问题；直接截断又可能丢失决策、
证据和工具协议。

### Task

在短任务继续使用完整原文、原始 session 可审计、不引入新依赖的条件下，让长任务可以
自动压缩并安全继续，同时避免将 LLM 摘要当作绝对可靠状态。

### Action

1. 用真实 API token 高水位加新增历史估算判断下一轮规模；
2. 设置 200K 自动触发、80K 目标和 240K 紧急门槛；
3. 只按完整工具轮次切分，并保留最近 40K 原文；
4. 用 forced tool call 生成版本化结构摘要；
5. 重新注入原始请求和代码生成的最新 Runtime Context；
6. 对论文 ID、决策标识和 evidence refs 做确定性完整性校验；
7. 校验失败最多重试一次，仍失败则停止；
8. 把压缩前后 token、成本和摘要写入 append-only JSONL；
9. 用边界测试和真实 qwen3.6-flash 受控长历史验证。

### Result

自动压缩已接入 Paper Copilot 主 loop，10 条专项测试及相关回归通过。真实模型受控评测
把 223,704 tokens 压缩到 22,859，减少约 89.8%，成本 ¥0.24626，所有预先声明的
决策和证据锚点均被保留。真实多轮 Composer A/B 未执行，因此不宣称最终质量无下降。

## 可能的面试追问

### 如何确保摘要不丢细节？

无法保证所有自然语言细节都不丢；压缩本身就是有损操作。系统保证的是更具体的安全边界：
原始请求重新注入、最新运行状态由代码重建、近期交互保留原文、关键标识做确定性校验、
原始 JSONL 永久保留。面试时主动承认这个边界比声称“无损压缩”更可信。

### 如果 LLM 两次都生成不合格摘要怎么办？

抛出错误并停止继续扩张上下文，不静默截断。240K 紧急门槛确保系统不会绕过失败的压缩
继续普通调用。

### 如何防止摘要伪造 evidence ref？

将摘要中形如 evidence ref 的标识与旧历史允许集合比较；新出现的引用直接判定为失败。
同时要求运行状态收集出的必需引用必须全部保留。

### 为什么估算器不是精确 tokenizer？

项目使用 API 返回的真实 input/cache token 作为上一轮锚点，本地估算只计算随后新增历史
的增量，并预留 200K 到 240K 的安全带。这样无需新增 tokenizer 依赖，也不会完全依赖
字符估算。

### 为什么压缩摘要也计入同一个成本预算？

压缩是完成用户任务所需的 LLM 调用。如果不计费，系统可能在预算耗尽后通过辅助调用继续
产生费用。成功、失败和 retry 都必须进入同一个 `CostTracker`。

### 与 RAG 有什么区别？

RAG 从外部知识库检索任务相关资料；这里压缩的是当前 Agent 的执行历史，包括决定、失败
路径、预算和工具状态。二者可以组合，但不能互相替代。

### 如果有更多时间，下一步做什么？

使用同一论文库、问题、模型和预算做多轮 Composer A/B：一组不压缩，一组降低测试阈值
确保触发压缩；比较 proposal checker、证据可解析率、状态一致性、成本和延迟，并重复运行
观察趋势。最后再做一次默认 200K 阈值的真实 canary。
